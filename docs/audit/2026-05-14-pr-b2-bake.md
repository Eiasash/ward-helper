## Pass / fail / fail-action conventions

Each station declares its fail-action up front so the disposition isn't litigated mid-bake:

- **BLOCKS PR-C** — station fails: do not open PR-C. File a follow-up workstream (v1.46.2 hotfix or B2 design revision) before any cleanup. The destructive surface is still reachable.
- **TRIAGE** — station fails: stop the bake, open a discussion with the workstream owner. Likely blocks PR-C, but not unconditionally.
- **RESIDUAL — RECORD** — station passes by design even when the visible outcome looks like a failure. The pass IS the bake catching a known hole; PR-C is not gated on this station closing the hole.

## Run order & state-reset matrix (READ BEFORE STATION 1)

> **The section order below is NOT the run order.** Stations are numbered for reference, not for execution sequence. Running them top-to-bottom will silently invalidate the bake.

**The trap:** Station 3 (backfill) is the only station that needs a *pre-seal plaintext* IDB state — plaintext patient/note rows that exist *before* the first v1.46.1 authed login. Every other station needs the *sealed* corpus that Station 3 produces. A tester who runs Station 2 before Station 3 either:

- finds Station 2 unsatisfiable on a fresh install (warm login seals nothing when there are no rows, but Station 2's procedure needs `≥1 sealed PHI row`), or
- seeds + seals data to make Station 2 work — which poisons Station 3, because the pre-seal plaintext state is now unreachable without a full IDB wipe and re-seed.

Either way the failure is latent: nothing errors, the tester just produces a bake that didn't exercise what it claims to. This is exactly the end-of-clinical-day failure mode — a tired tester following the page order, no real-time signal that setup was invalidated three stations ago.

**Run the stations in this order:**

1. **Station 1 — cold start.** State-agnostic. Do it first, on a genuinely virgin install, so the fresh-boot path is actually exercised.
2. **Station 3 — backfill.** The pivot. The *only* station needing pre-seal plaintext state — so it must run before anything that consumes sealed rows. Produces the shared sealed corpus the rest of the bake reuses.
3. **Station 4 — mixed-state read.** Chains directly off Station 3's all-sealed end state (+ the one manual plaintext insert).
4. **Station 5 — all-encrypted read.** Runs off the post-3 / post-4 steady sealed state. No reset needed if it follows 4 directly.
5. **Station 2 — password gate (2.1 → 2.2).** Reuses the sealed corpus from Station 3. Reset into `'locked'` state per its setup (wipe in-memory key, do NOT clear IndexedDB).
6. **Station 6a — wrong pwd + sealed PHI.** Same `'locked'` + sealed-corpus state family as 2.2; run it adjacent.
7. **Station 6c — password change vs sealed PHI.** Needs logged-in + sealed corpus. Non-mutating on pass.
8. **Station 6d — second device.** Needs the *original* device's corpus pristine and untouched — so it must run before the destructive Station 6b.
9. **Station 6b — wrong pwd + EMPTY stores. DESTRUCTIVE — must be last.** Its setup deletes every PHI row from the shared corpus, and it intentionally creates wrong-key-sealed orphans. Nothing can run after it without rebuilding from Station 3.

**State matrix:**

| Run # | Station | Entry state required | Reset action to reach entry state | Exit state | Destructive? |
|---|---|---|---|---|---|
| 1 | 1 — cold start | Virgin install / fresh private window, no prior storage | Force-quit browser or open clean private window | App booted; PHI stores empty | No |
| 2 | 3 — backfill | Fresh/v1.45.0 install; ≥1 plaintext patient + ≥1 note seeded **before** first v1.46.1 authed login; `Settings.phiEncryptedV7` null | If anything is already sealed: wipe the whole `ward-helper` IDB, then re-seed plaintext rows | All-sealed corpus; sentinel true; `phi_encrypt_v7` flag on | No (constructive) |
| 3 | 4 — mixed-state read | Station 3's all-sealed corpus + one manually-inserted plaintext row | Chains off run #2; just do the DevTools plaintext insert | All-sealed again (mixed-test row promoted on edit) | No |
| 4 | 5 — all-encrypted read | Post-3/post-4 all-sealed steady state | None if it follows run #3 directly | All-sealed; new rows from write exercises, still sealed | No |
| 5 | 2 — password gate | `'locked'`: sealed corpus on disk + auth profile in localStorage + no in-memory key | From the warm post-5 state: wipe in-memory key (close tab + clear cookies) **without** clearing IndexedDB | Warm; sealed corpus intact (on pass) | No (on pass) |
| 6 | 6a — wrong pwd + sealed | Same `'locked'` + sealed corpus as 2.2 | Re-establish `'locked'` state (same as run #5) | Sealed corpus intact, `N_post == N_pre` (on pass) | No (on pass) |
| 7 | 6c — password change | Logged-in correctly + sentinel true + ≥1 sealed row | Warm login with correct password | Sealed corpus byte-identical, `ct_pre == ct_post` (on pass) | No (on pass) |
| 8 | 6d — second device | Original device: pristine sealed corpus, untouched. Second device: confirmed-clean (no `ward-helper` IDB) | None on original device; on second device confirm `indexedDB.databases()` is empty | Original device pristine + unchanged; second device has its own independently-sealed rows | No (to original device) |
| 9 | 6b — wrong pwd + EMPTY stores | Sentinel true + **zero** sealed rows | **DESTRUCTIVE:** after the corpus has served runs 5–8, delete every row from patients/notes/roster via DevTools (keep Settings) | Wrong-key-sealed orphan rows (the residual demo); shared corpus gone | **Yes — terminal** |

**If you want to re-run the bake or preserve the corpus:** run Station 6b last *on a separate browser profile* seeded from a Station-3 repeat, rather than destroying the shared corpus. The template's setup assumes the shared corpus for simplicity; a separate profile is the non-destructive variant.

**Hard rule:** before starting any station, confirm its **entry state** via DevTools (IDB store shapes, `Settings.phiEncryptedV7`, sealed-vs-plaintext row shapes) — do not infer it from "the previous station should have left it this way." If the entry state isn't what the matrix says, STOP and reset; do not improvise a fix mid-station. A station run against the wrong entry state is not a partial result — it is a void result, and a void result that looks like a pass is the failure this matrix exists to prevent.

## Station 1 — cold start

**Goal:** App boots cleanly on the target device. No regressions from v1.45.0 cold-start behavior.

**Procedure:**
1. Force-quit the browser (or fresh private window with no prior cookies/storage).
2. Navigate to `https://eiasash.github.io/ward-helper/`.
3. If the homescreen PWA is installed, launch from homescreen instead.
4. Observe the boot sequence for ≤5 seconds.

**Pass criteria:**
- App reaches a rendered screen (login form, Unlock gate, or routes) within 5 seconds.
- No empty-rows flash. The "טוען..." placeholder may briefly render during the `'loading'` state — that's correct (per v1.46.0 fix). What MUST NOT happen: Routes rendering with an empty patient list visible, then flipping to Unlock or showing data.
- No JavaScript errors in console (if accessible via remote debug).
- No CSP violations visible.
- DecryptFailureBanner reads 0.

**Fail-action:** TRIAGE (likely blocks — cold-start regression is high-severity).

**Result:** ☐ pass  ☐ fail  ☐ partial

## Station 2 — password gate

**Goal:** Cold-start manual unlock works correctly for both correct-password (accepts) and wrong-password (rejects) paths against sealed PHI. This is the load-bearing surface the v1.46.1 hotfix landed on.

**Setup:**
1. Need to be in `'locked'` gate state: logged in (auth profile in localStorage) AND sentinel-true (sealed PHI on disk) AND no in-memory PHI key. Reach this by:
   - Log in with correct password (warm path establishes sentinel + sealed rows).
   - Verify sentinel via DevTools (mobile remote debug or desktop): open IDB → `ward-helper` → `settings` → `singleton` → confirm `phiEncryptedV7: true`. Confirm `patients` store has ≥1 row with shape `{id, enc}` (sealed) — NOT `{id, name, ...}` (plaintext).
   - Now wipe in-memory key without clearing localStorage: simulate a private-window reload. On mobile: close tab + clear cookies for the site BUT do NOT clear IndexedDB. (If IndexedDB also gets cleared the bake setup is invalid — restart from the warm-login step.)
   - Reopen the app. Should land on Unlock gate.

**Procedure (run sub-cases 2.1 and 2.2 in order, NOT in parallel):**

### 2.1 — correct password
1. Type the correct login password into the Unlock gate.
2. Tap "פתח".

### 2.2 — wrong password (run AFTER 2.1 fully passes, reset gate state first by re-establishing the locked state)
1. Type a known-wrong password (e.g., the correct password with one character changed).
2. Tap "פתח".

**Pass criteria:**

| Sub-case | Visible expected | Silent expected |
|---|---|---|
| 2.1 correct | Gate clears within ~1s, routes render, History shows existing patients | DecryptFailureBanner reads 0 |
| 2.2 wrong | Inline error "סיסמה שגויה — נסה שוב" renders. Field clears. Gate stays mounted. No routes accessible. | After dismissing and re-locking, log back in with the CORRECT password → DecryptFailureBanner STILL reads 0. (If 2.2 had silently sealed something under wrong key, the count would have jumped.) |

**Fail-action:** BLOCKS PR-C. The gate is the load-bearing surface; a fail here means the v1.46.1 fix is incomplete.

**Result 2.1:** ☐ pass  ☐ fail  
**Result 2.2:** ☐ pass  ☐ fail

## Station 3 — backfill

**Goal:** First-run PHI backfill on this device correctly seals plaintext rows, sets the sentinel, and flips the per-tab flag.

**Setup:** A fresh-install state OR a v1.45.0 install upgrading to v1.46.1. Need ≥1 patient + ≥1 note + ≥0 roster as plaintext rows in IDB BEFORE the first authed login on v1.46.1.

**Procedure:**
1. Confirm IDB state pre-login: patients/notes are plaintext shape (`{id, name, ...}` not `{id, enc}`).
2. Confirm `Settings.phiEncryptedV7` is null/absent.
3. Log in with the correct password.
4. Wait for the backfill to complete (typical: <1s for ward-scale data).

**Pass criteria:**
- Console breadcrumb `boot.phiBackfill` fires with `sealed > 0` (matches the count of plaintext rows pre-login).
- IDB state post-login: every patients/notes/roster row is sealed shape `{id, enc}` (for notes: `{id, patientId, enc}`).
- `Settings.phiEncryptedV7 === true` in IDB singleton.
- `localStorage.phi_encrypt_v7 === '1'`.
- Navigate to History — the previously-plaintext patient's notes still display correctly (key derivation + decrypt round-trip works end-to-end).
- DecryptFailureBanner reads 0.
- No data loss: pre-login patient count equals post-login patient count.

**Fail-action:** BLOCKS PR-C. Backfill failure means existing user data either didn't get sealed (defeats the workstream) or got mangled (data loss).

**Result:** ☐ pass  ☐ fail

## Station 4 — mixed-state read

**Goal:** The read seam correctly handles a store that contains both sealed and plaintext rows. This state is reachable in production via interrupted backfill (app closed mid-sweep, or cross-store partial commit).

**Setup:** Force mixed state. Easiest path:
1. After station 3 (all-sealed state), use DevTools to manually INSERT one plaintext patient row directly into IDB: `db.patients.put({id: 'test-mixed', name: 'Mixed-test', teudatZehut: '999999999', dob: '1950-01-01', room: null, tags: [], createdAt: Date.now(), updatedAt: Date.now()})`.
2. Confirm: patients store now has N sealed rows + 1 plaintext row.

**Procedure:**
1. Navigate to History. Both the previously-sealed patients AND the new plaintext `Mixed-test` patient should be visible.
2. Navigate to Today. Same expectation.
3. Open Settings → Database stats. Patient count should equal N+1.
4. Edit the mixed-test patient (e.g., add a tomorrow note via the Today screen). After save, confirm the row is now sealed `{id, enc}` (the write-side wrap fires on flag-on writes).
5. Edit one of the originally-sealed patients similarly. After save, still sealed.

**Pass criteria:**
- Both sealed and plaintext rows render in the UI.
- No crashes, no missing rows.
- DecryptFailureBanner reads 0.
- A write against a plaintext row promotes it to sealed shape (verified via DevTools post-save).
- A write against a sealed row keeps it sealed shape.

**Fail-action:** BLOCKS PR-C. Mixed state must work because interrupted-backfill produces it.

**Result:** ☐ pass  ☐ fail

## Station 5 — all-encrypted read

**Goal:** Steady-state operation when every row is sealed. The common case after backfill completes.

**Setup:** Post-station-3 state (or post-station-4 after the mixed-test row was promoted to sealed). Confirm via DevTools: every patients/notes/roster row is `{id, enc}` / `{id, patientId, enc}` shape.

1. Navigate `/` (Capture) — verify the screen renders.

## Station 6 — key divergence (added since the kickoff brief)

**Goal:** Verify the v1.46.1 probe-verify holds for the destructive failure mode. The original kickoff brief's 5 stations don't exercise this. Key divergence is the failure class where a wrong key gets set in memory and the system silently seals NEW data under it; the destructive mode is invisible at write time and surfaces only later as orphaned rows + DecryptFailureBanner increments.

**The silent-mode check (used in 6a, 6c, 6d):** after each scenario, log out + log back in with the KNOWN-CORRECT password → navigate to the screens that READ each affected PHI store (History reads notes, Today reads patients, Settings → DB stats reads roster) → record the DecryptFailureBanner count. If a wrong-key seal happened, the count will be > 0. **This count is the only signal that catches a false-accept.**

**Banner-only-fires-on-read gap (procedural — do not skip):** the DecryptFailureBanner increments only when the storage layer actually READS a row that fails to decrypt under the current in-memory key. A wrong-key-sealed row that's never read after the correct re-login would NOT increment the banner — the bake would false-pass by mechanical accident. So each sub-case below MUST explicitly navigate to History + Today + Settings → DB stats (in that order) after re-login, to force reads against all three PHI stores. The `N_pre`/`N_post` IDB row counts (DevTools-required) are the backstop for exactly this gap: they don't require a read to happen, and they surface row-count drift the banner can't.

### 6a — Wrong password at cold-start manual gate, against sealed PHI

**Setup:** Same as station 2.2 — `'locked'` state with ≥1 sealed PHI row on disk.

**Procedure:**
1. Open IDB DevTools, count sealed PHI rows. Record `N_pre` (e.g., 5 patients + 12 notes + 0 roster = 17 sealed rows).
2. Type a wrong password into the Unlock gate.
3. Observe response. DO NOT close the app.
4. Try to navigate via the URL bar to `/history`, `/today`, `/save`. (If gate is correctly mounted, these don't navigate away from the gate.)
5. Log out OR clear the AUTH localStorage key (force the auth state to flip).
6. Log back in with the CORRECT password.
7. Force reads against all three PHI stores: navigate to History (reads notes), then Today (reads patients), then Settings → DB stats (reads roster via the boot-time `ageOutRoster` plus an explicit fetch in stats). The banner only increments on read — skipping any one of these can mechanically false-pass the bake.
8. Read DecryptFailureBanner count.
9. Open IDB DevTools, count sealed PHI rows. Record `N_post`.

**Pass criteria:**
- Visible: `סיסמה שגויה — נסה שוב` error renders. Field clears. Gate stays mounted. No routes accessible during the wrong-key window.
- Silent count after correct re-login: DecryptFailureBanner reads **0**.
- `N_post == N_pre` (no new sealed rows from the wrong-key session, and no orphaned old rows).

**Fail-action:** BLOCKS PR-C. The v1.46.1 fix didn't hold; the destructive surface is still reachable.

### 6b — Wrong password at cold-start manual gate, against EMPTY PHI stores

**Setup:** Sentinel-true (`Settings.phiEncryptedV7 === true`) AND zero sealed PHI rows on disk. Reach this by: complete station 3, then via DevTools manually delete every row from patients / notes / roster stores (keep Settings).

**Procedure:**
1. Confirm IDB state: Settings.phiEncryptedV7 is true; patients/notes/roster stores are empty.
2. Trigger the locked-state setup (same as station 2 — wipe in-memory key without clearing IDB).
3. Type ANY password into the Unlock gate (deliberately wrong).
4. Observe: gate clears, routes render.
5. Use the app normally: create a NEW patient via capture, add a tomorrow note. (Both will seal under the WRONG key derived in step 3 — this is the demonstration.)
6. Log out. Log back in with the CORRECT password.
7. Force reads against all three PHI stores: navigate to History, then Today, then Settings → DB stats. The new patient + tomorrow note created in step 5 will get read here; their wrong-key seal can only surface via the banner once they're actually accessed.
8. Read DecryptFailureBanner count.
9. Open IDB DevTools, find the NEW patient created in step 5. Note: this row will be sealed under the wrong key from step 3.

**Pass criteria:** This station passes by design when the residual is observed.
- Step 4 succeeds (gate accepts) — this IS the residual.
- Step 7 onward: DecryptFailureBanner reads > 0 (the rows from step 5 orphaned because they were sealed under the wrong key). This is the EXPECTED demonstration of the residual.

**Fail-action:** RESIDUAL — RECORD. This station's purpose is to MEASURE the known hole, not close it. Record the observed banner count + the orphaned row count so the residual surface is documented per device + per OS in the bake archive.

**Closing the residual** is a separate Tier-2 design (server-side verification challenge) — out of scope for PR-C.

### 6c — Password change against sealed PHI

**Setup:** Logged in correctly, sentinel-true, ≥1 sealed row on disk.

**Procedure:**
1. Open IDB DevTools. Record `N_pre` (sealed row count). Pick one sealed row at random and screenshot its `enc.ciphertext` first 8 bytes — call this `ct_pre`.
2. Open DevTools Network tab + filter on `auth_change_password`.
3. Navigate Settings → change password form.
4. Enter the current password as the "old" field. Enter any new password as the "new" field.
5. Submit.
6. Observe the Network tab for the duration of the request.
7. Read the form's error message.
8. Whether the form succeeded or refused: log out, then log back in with the CORRECT password (the original one — if the rotation accidentally succeeded, "correct" is no longer accurate, and that itself is part of the bug-mode you're checking for). Force reads against all three PHI stores: History → Today → Settings → DB stats. The banner only increments on read; a silently-broken guard that orphaned existing sealed rows surfaces here.
9. Open IDB DevTools again. Record `N_post` and pick the same sealed row — screenshot its `enc.ciphertext` first 8 bytes as `ct_post`.

**Pass criteria:**
- Visible: the change-password form renders an explicit error message containing the Hebrew phrase `שינוי סיסמה לא נתמך` (or the equivalent `phi_rotation_not_supported` error code).
- Silent (Network tab): NO request fires to `auth_change_password`. The guard fires BEFORE the RPC.
- Silent (banner check after the forced reads in step 8): DecryptFailureBanner reads 0. If the guard had silently failed and the rotation orphaned existing PHI, the banner would have incremented on read.
- Silent (IDB invariants): `N_post == N_pre`. `ct_pre == ct_post`. Sealed rows are byte-identical before and after.

**Fail-action:** BLOCKS PR-C. The rotation guard isn't working; the original destructive failure mode (password change → orphan all sealed PHI) is reachable.

**Result:** ☐ pass  ☐ fail

### 6d — Same account on a second device, fresh local salt

**Setup:** A SECOND device (or fresh browser profile) NEVER used with this account. Original device has sealed PHI from the same account. Note: cross-device PHI sync is not shipped (cloud-side encrypted-blob runtime is a separate pending workstream), but this is current code in a scenario already reachable in production — multi-PWA users have one account that can land on a second device today.

**Procedure:**
1. **Original device — pre-snapshot:** open IDB DevTools, count sealed PHI rows. Record `N_pre_origin`. Read DecryptFailureBanner count (should be 0). Note current sealed-row ciphertext sample (first 8 bytes of any one row's `enc.ciphertext`) — call it `ct_pre_origin`.
2. **Second device — confirm clean state:** `indexedDB.databases()` shows no `ward-helper` DB or an empty one.
3. **Second device — login:** navigate to the app on the second device. Log in with the same username + password as the original device. Wait for boot completion.
4. **Second device — post-login state:** open IDB DevTools. Confirm `Settings.phiSalt` is present and is a fresh random 16-byte value (independent of the original device — both are random).
5. **Second device — read attempt:** navigate to History (reads notes), Today (reads patients), Settings → DB stats (reads roster). The forced reads are essential: if the second device's locally-derived key somehow encountered sealed data from another source (cloud-restored, accidentally pulled), the banner would increment here. Otherwise zero rows to read.
6. **Second device — write attempt:** create a NEW patient via capture. Add a tomorrow note. Save.
7. **Second device — verify second-device write:** open DevTools, confirm the new patient row is sealed `{id, enc}` shape. The encryption is under THIS device's locally-derived key (password + second-device-salt).
8. **Second device — banner check:** DecryptFailureBanner still reads 0 on this device (no orphans here either).
9. **CROSS-DEVICE — switch back to ORIGINAL device:** force a reload (close tab, reopen, or hard-refresh). Wait for boot.
10. **Original device — force reads:** navigate to History → Today → Settings DB stats. The banner must increment if anything cross-device-orphaned was introduced.
11. **Original device — post-snapshot:** open IDB DevTools, count sealed PHI rows. Record `N_post_origin`. Pick the same row sampled in step 1 — record `ct_post_origin`. Read DecryptFailureBanner.

**Pass criteria:**
- Step 3 succeeds without crash or wedge.
- Step 4: local salt on second device is non-empty and (almost certainly) byte-different from original device's `phiSalt`.
- Step 5: second device's banner reads 0 (no unexpected sealed data).
- Step 7: second device's new patient is sealed under its own key — `{id, enc}` shape.
- Steps 10–11: original device unchanged. `N_post_origin == N_pre_origin`. `ct_post_origin == ct_pre_origin` (byte-identical). DecryptFailureBanner on original device reads 0. No cross-device contamination.

**Fail-action — SPLIT by outcome class (set by worst credible outcome, not scenario name):**

| Failure observed | Disposition |
|---|---|
| Cross-device wrong-key seal: second-device write somehow lands as a sealed row on the original device's IDB under a divergent key | **BLOCKS PR-C** — same destructive failure mode v1.46.1 closed for the cold-start manual path. If 6d turns this up, the hotfix left a second door open. |
| Orphan on original device: `N_post_origin != N_pre_origin`, `ct_post_origin != ct_pre_origin`, or banner increments on original after second-device login | **BLOCKS PR-C** — data-loss-class even if the mechanism is different from 6a/6c. |
| Original device's persisted login password gets wiped/rotated by the second-device login (would leave original in the residual-6b state) | **BLOCKS PR-C** — destructive precursor. |
| Second-device login crash, UI wedge, hang, clean refuse with no destructive consequence | **TRIAGE** — flag for the auth-flow workstream, doesn't necessarily block PR-C unless the fix surface overlaps. |
| Benign re-derivation that just fails cleanly (e.g., login error on second device with no data effect) | **TRIAGE** |

Default disposition for ambiguous failures: **BLOCKS** (fail-closed, same bias as everywhere else in this template).
