# v1.46.1 hotfix — cold-start unlock-gate probe-verify

**Shipped:** 2026-05-14 (same day as v1.46.0 / PR #169).
**Branch:** `claude/term-phi-unlock-verify-1-46-1`.

## What this hotfix fixes

v1.46.0 shipped the local PHI-at-rest layer. A static trace immediately after merge surfaced a destructive failure mode in the cold-start `Unlock.tsx` gate:

1. User on v1.46.0 with `Settings.phiEncryptedV7 === true` (already-migrated) loses their persisted login password — private-window reload, profile reset, Settings IDB partial wipe.
2. Cold-start renders the Unlock gate (sentinel-true + no in-memory key + logged-in).
3. User mistypes their password into the gate.
4. Pre-1.46.1 behavior:
   - `attemptPhiUnlockWithPassword(WRONG)` derives a wrong key from `WRONG + persistedSalt`.
   - `setPhiKey(wrong_key)` is set in memory.
   - The sentinel-gated backfill SKIPS (sentinel is already true).
   - `_deriveAndBackfill` returns `{kind: 'ok'}`.
   - Gate clears, routes render, `hasPhiKey()` is true.
5. Reads of existing sealed rows return null (wrong-key decrypt fails) — `DecryptFailureBanner` increments. Reads alone are recoverable.
6. **The destructive step:** the user takes a write action that goes through one of the pure-writer paths — `putPatient` (e.g., admission via `upsertPatientByTz` from `saveBoth`), `putNote`, `setRoster`. Those paths don't read-decrypt first; they call `wrapPatientForWrite`/`wrapNoteForWrite`/`wrapRosterForWrite` which seal with the WRONG KEY now in memory.
7. The user eventually logs out + back in with the correct password. Correct key now in memory. The wrong-key-sealed rows from step 6 orphan permanently. They are NEW clinical data — new admissions, new notes, new roster imports created during the wrong-key session. Not recoverable.

The pre-1.46.1 `_stagedPatientUpdate` helper wasn't the leak: it reads-then-decrypts and throws on decrypt-failure before writing. The leak was the pure-writer paths, which seal without verifying against existing data.

## The fix

`_deriveAndBackfill` in `src/auth/phiUnlock.ts` gains an `verify: boolean` parameter (default false). The cold-start manual path `attemptPhiUnlockWithPassword` passes `verify=true`; the warm-login path `attemptPhiUnlock` stays at default false because its password came through server-bcrypt verification.

When `verify=true`, before running backfill:

1. Pull up to 3 sealed rows from each of patients/notes/roster (9 max samples).
2. `isEncryptedRow`-gate every candidate so plaintext rows in mixed-state storage don't enter the probe set.
3. Direct `unsealRow` call (NOT `decryptRowIfEncrypted`) on each sample so probe failures don't increment the user-facing `DecryptFailureBanner` counter — probe failures are diagnostic, not user-visible incidents.
4. One successful decrypt = `'verified'`, return early to backfill.
5. Zero samples passed `isEncryptedRow` = `'no-sealed-rows'` — accept (DOCUMENTED RESIDUAL — see below).
6. All samples failed AND at least one was sealed = `'wrong-key'` — `clearPhiKey()` and return `{kind: 'wrong-password'}`.

Multi-row sampling guards against false-reject when a single row is genuinely corrupt (damaged ciphertext, not wrong-key). The probe's contract: "key is correct iff at least one sealed row decrypts." This is the right bias for fail-closed.

## Documented residual — empty-PHI cold-start

The probe cannot protect installs in the state: sentinel-true + zero sealed rows on disk. With nothing to verify against, the probe returns `'no-sealed-rows'` and the password is accepted.

This is a narrow population — it requires:
- The install completed PHI backfill in a prior session (sentinel-true), AND
- All sealed rows have since been deleted (e.g., user manually wiped patients/notes/roster but kept the Settings record), AND
- The user lost their persisted login password (so the cold-start gate renders), AND
- The user types a wrong password.

In this state, the wrong key sets in memory, the user uses the app, and any new writes seal under the wrong key — destructive only IF the user later logs in with the correct password from another source and the wrong-key rows then orphan.

**Mitigation:** Bake station 6 measures this case explicitly. The destructive surface is far smaller post-1.46.1 (narrows from "every install with sealed rows" to "the empty-but-sentinel-true population only"). Eliminating it entirely requires a server-side verification challenge (e.g., a derive-and-encrypt-known-plaintext server round-trip) — out of scope here, candidate for a future Tier-2 design.

## Outcome surface change

`PhiUnlockOutcome` gains a `'wrong-password'` variant:

```ts
export type PhiUnlockOutcome =
  | { kind: 'ok'; report: PhiBackfillReport }
  | { kind: 'no-user' }
  | { kind: 'no-password' }
  | { kind: 'already-unlocked' }
  | { kind: 'wrong-password' }    // NEW
  | { kind: 'backfill-failed'; error: Error };
```

Consumer surfaces re-audited for default-fallthrough hazard:
- `src/ui/screens/Unlock.tsx::onSubmit` — explicit `'wrong-password'` branch added; pre-1.46.1 message `"סיסמה שגויה — נסה שוב"` lived under `'backfill-failed'` (which was wrong — backfill-failed is a genuine system error, not a password signal). Now: `'wrong-password'` → `"סיסמה שגויה — נסה שוב"`; `'backfill-failed'` → `"שגיאה במערכת: <error.message>"`.
- `src/ui/App.tsx` boot + auth subscriber — calls warm-path `attemptPhiUnlock` only; will not see `'wrong-password'`. Breadcrumb log treats `outcome.kind` as opaque string and tolerates new variants.

## Bake station 6 (added since the kickoff brief)

The kickoff brief listed 5 bake stations (cold start / password gate / backfill / mixed-state read / all-encrypted read). None of those exercise key divergence — the destructive failure mode. Station 6 closes that gap:

**Station 6 — key divergence.**

Sub-cases:
- (6a) Wrong password into cold-start gate against sealed PHI → `'wrong-password'`, gate stays mounted, `hasPhiKey()` false, no key in memory.
- (6b) Wrong password into cold-start gate against EMPTY PHI stores (sentinel-true, zero sealed rows) → accept (DOCUMENTED RESIDUAL; bake records this is by design).
- (6c) Password rotation via `changePasswordWithReencrypt` against sealed PHI → `phi_rotation_not_supported` refused, sealed rows untouched on disk.
- (6d) Two-device behavior (same Supabase account on second device with freshly-derived local salt): not in scope until cloud-side encrypted-blob runtime ships. Verify the bake setup doesn't accidentally hit this case.

Pass criteria: stations 6a + 6c block the destructive paths; station 6b records the known residual.
